## 三、使用阶段
目前使用阶段推理速度无法达到预期，主要原因如下。

### 1. 频繁加载模型与初始化进程
以 7.78MB 神经网络为例，迭代 1 万次，每次查询 3 万点：  
单次迭代需加载模型与初始化进程 9 次，用时约 0.17s，全程约 **1700s**，与 CPU 推理耗时同级甚至更多。  
需对调用方式进行优化，减少初始化次数。

### 2. C++ 平台推理速度不如原生 PyTorch
cpp 端 CPU 推理比 PyTorch 慢约 30%–40%。

### 3. 大规模高频推理下 CPU 性能分析与 GPU 加速

### 当前推理性能瓶颈的根源不是模型，而是任务规模与计算密度。

### 模型规模
- 输入维度：9  
- 输出维度：9  
- 隐藏层：4 层  
- 每层神经元：532  

该结构属于典型小型 MLP，单次推理 FLOPs 在百万级以内，CPU 单次推理本身毫秒级，非常快。

### 性能瓶颈
- 动力学仿真迭代：上万次  
- 每次迭代查询：1w–5w 点对  
- 总推理调用量：**数亿级**

即便单次推理几十微秒，整体耗时仍会累积至分钟到小时级。

CPU 的并行度、吞吐量和内存带宽均不足以支撑如此密集计算需求；

神经网络推理本质上是矩阵乘法，属于高度并行任务。
因此，在相同代码质量下，**GPU 推理速度通常比 CPU 快至少一个数量级**。  
当推理次数达到“亿级”时，CPU 的串行瓶颈无法避免，必须使用 GPU 来进行大规模并行矩阵运算。

## 四、总结

当前柔度数据的高效压缩存储与快速模型训练流程已基本实现。  
后续工作包括：

- 通过 GPU 加速进一步提升大规模、高频场景下的推理吞吐率；
- 优化 C++ 调用方式，减少模型加载与初始化开销；
- 通过批量推理、并行推理等方式提升整体计算效率。
